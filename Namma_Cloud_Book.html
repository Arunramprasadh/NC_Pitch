The Sovereign Imperative

The geopolitical moment has arrived. Sovereign cloud is no longer a philosophical preference — it is a regulatory requirement, a national security posture, and an economic necessity.

The World Has Changed

For most of the 2010s, "cloud" meant AWS or Azure. Cheap, scalable, globally available. The tradeoffs — data crossing borders, dollar-denominated costs, dependence on foreign infrastructure — felt abstract. They no longer do.

The Russia-Ukraine war demonstrated that global payment rails can be severed overnight. US-China tariff escalations showed that technology supply chains are weaponisable. India's own experience with data localisation debates, the Aadhaar controversy, and NPCI's insistence on data sovereignty have all pointed in one direction: critical national infrastructure must be controlled nationally.

🌍

Russia-Ukraine, Iran tensions, and US-China decoupling have made dependency on foreign cloud a sovereign risk — not just a technical one.

⚖️

Regulatory Pressure

💱

FX Economics

Dollar-denominated cloud bills have not followed Moore's Law. Banks earning in INR face 15–30% cost inflation purely from currency exposure.

MeitY's March 2026 Directive

In a March 2026 office memorandum titled Guidelines for Cloud Selection Framework, the Ministry of Electronics and Information Technology directed all ministries and departments to classify their data into three categories before selecting any cloud service:

🔴 Top Secret / Secret

Strictly not permitted to be hosted on commercial cloud platforms at all. Must remain on sovereign, air-gapped infrastructure.

🟡 Category A — Sensitive National Data

Includes Aadhaar, PAN, e-Courts, UPI data, passport records, railways, tax systems, voter ID (central); land records, revenue courts, treasury (state). Must be hosted on government cloud services or sovereign-certified providers. No commercial hyperscalers.

🟢 Category B — Official Use Data

RTI-disclosable data, welfare scheme data, public grievances. May be hosted on central/state cloud services or MeitY-notified sovereign providers.

The implication for banks is unambiguous: UPI transaction data, card data, Aadhaar-linked KYC — everything that makes modern banking work — cannot legally sit on AWS, GCP, or Azure. The only compliant path is a sovereign cloud certified by MeitY, or an on-premise infrastructure operated in India.

India's RBI Position

The Reserve Bank of India has long maintained that payment system data must be stored exclusively in India. The 2018 circular on Payment System Data Storage made this explicit for UPI and card networks. The 2023 Master Directions on Outsourcing extended this to cloud service providers, requiring banks to ensure data sovereignty, right to audit, and exit clauses.

💡 The Compliance Trap

Banks are caught in a bind: modern cloud-native architecture requires hyperscalers, but RBI mandates on-premise or sovereign hosting. Legacy on-prem is too brittle for modern workloads. The only escape is a sovereign cloud that is actually cloud-native — which is exactly what Namma Cloud is.

The Window Is Narrow

Banks that move to sovereign cloud in the next 18–24 months will build on modern, AI-powered infrastructure that gives them a 10-year cost and capability advantage. Banks that wait will face forced migrations under regulatory pressure, with less choice, less time, and higher risk.

The sovereign moment is here. The question is not whether to move, but with whom.

How InfraSwitch Works

Every infrastructure operation in InfraSwitch is a workflow — a typed, auditable state machine that progresses through pre-check, execute, and validate phases:

-- Every infrastructure change is a ReleaseTracker state machine
CREATED → IN_PROGRESS → UNDER_REVIEW → APPROVED
        → START_RELEASE
        → [Stage 1 .. Stage N execution]
        → STAGGER → MONITORING → POST_MONITORING
        → STABILIZE → COMPLETED

-- On failure: automatic rollback
        → ROLLBACK → REVERT_CHANGES → UNLOCK

Every stage has three phases: PRE_TEST (validate preconditions), EXEC (make the cloud API call), and VALIDATE (poll until confirmed). This three-phase model handles eventual consistency correctly — something most operations tools get wrong.

Namma DSL — Infrastructure as Truth

Namma DSL is a Domain Specific Language built in Rust for declaring and managing datacenter infrastructure as code. Its founding philosophy: invalid infrastructure configurations should not compile.

This is not Infrastructure as Code in the Terraform sense. This is Infrastructure as Type-Safe Code — where the compiler enforces correctness before anything touches production. A misconfigured cable type, a rack capacity violation, an invalid CPU configuration — all caught at compile time.

Reliability by Design

99.999% availability is not an aspiration at Juspay — it is an engineering requirement. Every architectural decision in Namma Cloud is made through the lens of: "What happens when this fails?"

The Ten Tenets

Juspay's Reliability Principles are not a marketing document. They are the operating philosophy that governs every infrastructure decision, every deployment, and every incident response.

Abundant Redundancy, Active-Active, Instant Failover

No single point of failure at any layer. Every critical component has a symmetric redundant counterpart, actively processing requests. Failover is instant and automatic.

Critical Path — 100X Strong, Utterly Simple

The critical path (payment processing, UPI routing, authentication) must be impossibly simple and impossibly strong. Complexity is the enemy of reliability.

Reduce Blast Radius

Failures should be contained. Merchant-level sharding, namespace-level isolation, circuit breakers between services — every boundary limits the damage of any single failure.

Auto Anomaly Detection, Auto RCA

Failures should be detected and diagnosed before humans are paged. Shifu and Shifu-2.0 provide millisecond anomaly detection and automated root cause analysis.

Dynamic Model-Based Capacity Allocation

Capacity must be allocated based on traffic models, not historical averages. Styrofoam ensures zero bottlenecks by predicting demand and provisioning ahead of it.

Mandatory Staggering, A/B Testing & Rollback for All Changes

No change — code, config, or infrastructure — goes directly to 100% of traffic. Every change is staggered, monitored, and ready to roll back in seconds.

Testing the Limits — Load, Failure, Chaos Tests in Production

We chaos test in production. Load tests validate scale claims. Failure injection tests validate failover assumptions. We test limits before customers find them.

RCA/CoE Process Done with Rigour

Every incident is a Root Cause Analysis. Every RCA is a Culture of Excellence opportunity. No incident is closed without a systemic fix in the critical path.

Set Ambitious SLAs/SLOs — Constantly Improve

99.999% is not a target we declare on slides. It is an SLO we measure in dashboards, review in post-mortems, and improve every quarter.

Org-Wide Collective Ownership

Reliability is not the SRE team's problem. It is every engineer's problem, every product manager's problem, every CXO's problem. The culture reinforces it daily.

The Yin and Yang of Reliability

Juspay's reliability philosophy recognises a fundamental tension in distributed systems: the same properties that make a system flexible and scalable (decentralisation, diversity, complexity) are the properties that make it fragile. Reliability requires holding both sides simultaneously.

⚫ Critical Path (Yin)

Unified · Centralised · Simple · Stable · Certain · Very Strong. The critical path — failover switches, load balancers, rate limiters, authentication — must be narrow, strong, and never fail.

⚪ Abundance Side (Yang)

Diversity · Redundancy · Distribution · Decentralisation · Complexity · Possibility. The system's resilience comes from abundant redundancy outside the critical path.

💡 The Golden Rule

Every RCA must ask: "Why did the critical path not catch this?" If the CP caught it, it was handled. If it didn't, the CP needs to be strengthened. This recursive improvement loop is why Juspay's uptime keeps improving.

Secret Management: OpenBao + IMDS

One of the most common — and most dangerous — security anti-patterns in cloud infrastructure is storing credentials on disk or in environment variables. Every bank that has had a credential leak knows this cost.

Namma Cloud uses OpenBao (an open-source, auditable fork of HashiCorp Vault) combined with IMDS (Instance Metadata Service) for secure VM secret access. No credentials ever touch disk. Secrets are fetched at runtime via instance identity, with full audit trails.

🔐 How It Works

A VM that needs a database password does not have the password in its config file. Instead, it presents its IMDS-provided identity token to OpenBao, which validates the identity and returns the secret — temporarily, with an expiry. The secret is never stored. The identity is unforgeable. The audit trail is complete.

Physical DC setup, Namma DSL baseline declaration, network and storage commissioning. Zero-touch provisioning. OpenBao secret management deployed. InfraSwitch connected to the new environment. No production traffic moved yet.

02
Observability Layer
Months 2–5 (overlapping)

Namma Observe deployed to monitor both the existing infrastructure and the new Namma Cloud environment. Anomaly detection baselines established. Alert consolidation — one Slack channel, one mobile app, one dashboard. Shadow mode: Namma Cloud receives traffic but does not process it yet.

03
Non-Critical Workload Migration
Months 4–8

Non-critical workloads — analytics, batch processing, reporting, internal tools — migrated first. Autopilot blue-green deployments. Styrofoam validates cost projections against actuals. Team builds operational muscle before touching production payments.

04
Payment Stack Migration
Months 8–12

UPI, cards, and netbanking workloads migrated with parallel-run strategy. Namma Cloud receives shadow traffic, results compared in real time. Staggered cutover: 5% → 25% → 50% → 100% with A/B statistical validation at each stage. Rollback available at any point.

05
Full Sovereignty
Months 12–18

Complete exit from commercial hyperscalers. All data on Indian sovereign infrastructure. Single pane of glass for all operations. Namma DB tiering in production. Full Juspay product integration live. Hyperscaler contracts terminated.
