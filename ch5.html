6  Core Capabilities

Chapter 5

Five purpose-built components form the heart of Namma Cloud. Each solves a distinct problem that banks face today. Together, they form a complete infrastructure operating system.

6.1 InfraSwitch — The Operations Brain

InfraSwitch is the central nervous system of Namma Cloud. Every infrastructure change — a deployment, a scaling event, a configuration update, an alert — flows through InfraSwitch's workflow engine.

Built in Haskell with 722 source files, InfraSwitch manages 142+ automated infrastructure workflows across Kubernetes, OpenStack, AWS (for hybrid setups), and GCP. It replaces the 10+ tools most banks use today with a single, unified command centre.

6.1.1 🎛️ Unified Dashboard

Single pane of glass for cluster setup, Kubernetes management, OpenStack operations, scaling, and configuration across all data centres.

6.1.2 🔄 Zero-Downtime Releases

Blue-green deployments with automated A/B testing and staggered rollouts. Every change is monitored, with automatic rollback on anomaly detection.

6.1.3 🔒 Four-Dimensional RBAC

Role × Workflow × Resource Pattern × Cloud Provider — fine-grained access control with full audit trails for every infrastructure change.

6.1.4 🛡️ Crash-Safe by Design

Persistent poll-driven state machine. If a process dies mid-execution, the system resumes from the last checkpoint. At-least-once execution, always.

6.1.5 How InfraSwitch Works

Every infrastructure operation in InfraSwitch is a workflow — a typed, auditable state machine that progresses through pre-check, execute, and validate phases:

-- Every infrastructure change is a ReleaseTracker state machine
CREATED → IN_PROGRESS → UNDER_REVIEW → APPROVED
        → START_RELEASE
        → [Stage 1 .. Stage N execution]
        → STAGGER → MONITORING → POST_MONITORING
        → STABILIZE → COMPLETED

-- On failure: automatic rollback
        → ROLLBACK → REVERT_CHANGES → UNLOCK

Every stage has three phases: PRE_TEST (validate preconditions), EXEC (make the cloud API call), and VALIDATE (poll until confirmed). This three-phase model handles eventual consistency correctly — something most operations tools get wrong.

6.2 Namma DSL — Infrastructure as Truth

Namma DSL is a Domain Specific Language built in Rust for declaring and managing datacenter infrastructure as code. Its founding philosophy: invalid infrastructure configurations should not compile.

This is not Infrastructure as Code in the Terraform sense. This is Infrastructure as Type-Safe Code — where the compiler enforces correctness before anything touches production. A misconfigured cable type, a rack capacity violation, an invalid CPU configuration — all caught at compile time.

Layer 0

Physical (Metal)

Dell R740/R750, HPE DL385 servers · Arista 7050X3, Mellanox SN2700 switches · 42U racks with power budgets

Layer 1

Operating System / Hosts

Ubuntu / RHEL / Arista / SONiC · Network interface configuration · Zero-touch provisioning via MAAS / Redfish

Layer 2

Services

OpenStack control plane / compute · Ceph storage clusters · Kubernetes nodes · BGP routing (FRR)

6.2.1 The Namma DSL Workflow

-- Step 1: Declare desired state in Rust DSL
spec! {
    Layer0 {
        racks:   [rack_mumbai_1, rack_mumbai_2],
        servers: [dell_r750_01..dell_r750_24],
        switches: [arista_7050x3_top_of_rack],
        links:   [/* typed: Sfp28Cage → DAC cables only */]
    }
    Layer1 {
        hosts:   [/* Ubuntu 22.04 on each server */]
        network: [/* management + data VLANs, separate */]
    }
    Layer2 {
        clusters: [ceph_storage, openstack_control, k8s_prod]
        bgp:      [/* ASNs auto-allocated, neighbors auto-discovered */]
    }
}

-- Step 2: Compile → validates topology invariants
-- Step 3: diff(old_state, desired_state) → deterministic plan
-- Step 4: InfraSwitch executes the plan
-- Step 5: Observe and track reconciliation

✅ Type Safety in Practice

If you try to connect an RJ45Port to an Sfp28Cage, the code won't compile. If you try to fill a 42U rack beyond its power budget, the code won't compile. If you try to deploy a routable network node without a configured host, the code won't compile. Every constraint in your datacenter design is enforced before a single command is run.

6.3 Autopilot — Zero-Impact Deployments

Autopilot automates the riskiest moment in any software operation: the release. Using blue-green deployments, staggered rollouts, and integrated A/B testing, Autopilot makes every code change a non-event — even for critical payment services.

6.3.1 🟦🟩 Blue-Green Deployments

Two identical environments. New version deployed to green while blue serves live traffic. Instant cutover. Instant rollback.

6.3.2 📊 A/B Statistical Testing

Every release is monitored with binomial, Poisson, and t-test statistical engines. Releases that degrade metrics are automatically aborted before reaching full traffic.

6.3.3 🎯 Staggered Rollouts

5% → 25% → 50% → 100%. Each stage monitored. Automated continue / warn / abort decisions. No human needed in the critical path.

6.4 Styrofoam — Cost & Capacity Intelligence

Styrofoam is Namma Cloud's predictive capacity planning and cost optimisation engine. It does what every bank wishes its cloud billing dashboard did: tell you in advance what you'll need, and tell you right now what you're wasting.

6.4.1 📈 Predictive Scaling

Uses time-series decomposition (trend + seasonality + residual) to forecast transaction patterns. Scales infrastructure before the load arrives, not after.

6.4.2 🗑️ Wastage Detection

Runs daily. Compares provisioned vs actual usage across EC2, RDS, ElastiCache, and bare metal. Flags over-provisioning with actionable right-sizing recommendations.

6.4.3 💳 Cost Calculator

Multi-cloud pricing engine supporting EC2, EBS, RDS, ElastiCache, Load Balancers, NAT Gateways. Compare AWS vs GCP vs Namma Cloud per workload.

6.4.4 🎯 ML Recommendations

Statistical analysis of p50/p95/p99 metrics with confidence scores. Detects growth trends. Recommends optimal resource sizes with safety headroom.

6.5 Namma Observe — Full-Stack Visibility

Every layer of Namma Cloud — from bare metal CPU utilisation to UPI transaction success rates — is observable through a single, unified monitoring platform. Not just dashboards. Intelligent, automated observability.

  • Infrastructure Metrics — CPU, memory, disk, network for every machine. Real-time and historical.
  • Application Metrics — Business metrics frameworks for payment success rates, latency, throughput per service.
  • ML Anomaly Detection (Shifu 2.0) — RandomForest, Prophet, IsolationForest models trained on your traffic patterns. Catches anomalies before they become outages.
  • Automated RCA — Graph-based root cause analysis with temporal causality. When an alert fires, the dashboard of the most anomalous metrics is automatically generated and posted to your Slack.
  • Mobile Alerts (Jusping) — Firebase push notifications, critical alert sounds, per-alert mute/acknowledge. Your on-call engineer knows about issues before customers do.
  • Namma DB Tiering — Database storage tiered across SSD → HDD → Tape, with automated data promotion/demotion based on access patterns.
